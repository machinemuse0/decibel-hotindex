# Dataset Layout

Last updated: 2026-05-27

Decibel HotIndex treats datasets as first-class build artifacts. Mainnet chain data is recorded once into an immutable raw archive, normalized locally, converted into deterministic query corpora, and replayed into storage backends. Benchmark runs consume local datasets only.

## Rule

Benchmark runner must not call Aptos gRPC, download chain data, or generate random non-hit keys. Online data access belongs to `decibel-dataset record`; benchmark uses a dataset directory, query corpus, and materialized DB path.

For real data, mainnet bounded recording is the primary path. Testnet is only a fallback for auth/parser coverage when mainnet access is blocked.

The raw archive is the source of truth. RocksDB and ToplingDB materialization must both replay from the same saved raw/normalized dataset instead of pulling from Aptos twice.

## Dataset Tiers

### P0: Synthetic Fixture Dataset

Purpose: CI, storage conformance tests, parser adapter smoke tests, and local benchmark smoke runs.

Scale: 1k to 10k normalized events.

Source: deterministic Decibel-like synthetic rows.

Benchmark use: not for external performance claims.

### P1: Mainnet Bounded Dataset

Purpose: primary realistic benchmark and grant material.

Source: Aptos Transaction Stream mainnet bounded range.

Benchmark use: publishable only with manifest sha256 and checksum pass.

### P2: Testnet Historical Dataset

Purpose: fallback event coverage and parser/schema compatibility.

Source: Aptos Transaction Stream testnet bounded range.

Benchmark use: engineering validation, not final public report.

### P3: Mainnet Rolling Dataset

Purpose: read-under-ingest, ingest lag, and live demo.

Source: near-real-time Transaction Stream from a checkpoint.

Benchmark use: live behavior; do not mix with bounded serving benchmark claims.

### P4: Synthetic Amplification Dataset

Purpose: stress and capacity planning when real Decibel data volume is too small.

Source: expanded keyspace/timestamp/account/market/builder distributions derived from real events.

Benchmark use: must be explicitly labeled synthetic amplification.

## Storage Formats

### Raw Transactions

Default raw format for real mainnet/testnet recording:

```text
length-delimited Aptos Transaction protobuf messages, compressed with zstd
file extension: .pb.zst
```

Rationale:

- closest practical representation to the gRPC payload
- much smaller than JSON
- can be replayed repeatedly into RocksDB and ToplingDB
- parser/schema changes can re-normalize locally without another chain download

Raw files should be chunked by version range. Each chunk is written to a temporary filename, flushed, hashed, then atomically renamed.

### Normalized Rows

Default normalized format for early MVP:

```text
ndjson.zst
```

Rationale:

- easy to inspect and diff
- still much smaller than uncompressed JSON because zstd compresses repeated field names and addresses well
- enough for P0/P1/P2 MVP scale

For larger P3/P4 datasets, add an optional binary replay format such as length-delimited MessagePack or bincode plus zstd. The manifest must record the encoding for every artifact.

### Audit Samples

Small uncompressed `.ndjson` samples are allowed under `reports/` or `samples/` for debugging. They are not the canonical raw archive.

## Commands

Target command shape:

```bash
rtk cargo run -p decibel-dataset -- synthetic --out datasets/fixtures/synthetic_smoke
rtk cargo run -p decibel-dataset -- fixture --out datasets/fixtures/decibel_fixture --events 100
rtk cargo run -p decibel-dataset -- normalize --input datasets/fixtures/decibel_fixture/raw/fixture_events.jsonl --out-dir datasets/fixtures/decibel_fixture/normalized --format fixture-jsonl
rtk cargo run -p decibel-dataset -- normalize --input <raw> --out-dir <dataset>/normalized
rtk cargo run -p decibel-dataset -- build-query-corpus --events <events> --out-dir <dataset>/queries --seed 42
rtk cargo run -p decibel-dataset -- replay --engine rocksdb --events <events> --db-path <dataset>/materialized/rocksdb
rtk cargo run -p decibel-dataset -- record --network mainnet --endpoint https://grpc.mainnet.aptoslabs.com:443 --start-version 4365621793 --end-version 4366621793 --out-dir <dataset>/raw --raw-format protobuf-zstd
```

## Query Corpus Rules

- `point_tx_versions` must be sampled from real transaction versions in the dataset.
- `multi_get_tx_versions` must batch real transaction versions in the dataset.
- `market_fill_scans` must use market IDs present in fills/activity rows.
- `account_fill_scans` must use accounts present in rows.
- `builder_code_scans` must use builder addresses present in attribution rows.
- `builder_code_volumes` must use builder addresses present in attribution rows.
- `mixed_dashboard` must combine hit-capable primitive query records according to the configured workload mix. It is a workload file, not a requirement that records use `query_kind=mixed_dashboard`.

Random non-hit keys are allowed only in explicit negative-query workloads and must be reported separately.

## Reproducibility Rules

- Every benchmark report references `dataset_id`, not just a local path.
- Manifest sha256 must cover raw, normalized, and query corpus files.
- Recording supports resume from `last_success_version` once real gRPC support exists.
- RocksDB and ToplingDB benchmark databases must be materialized from the same raw/normalized dataset.
- The raw archive is immutable. If normalization logic changes, produce a new normalized artifact and update manifest/parser metadata instead of overwriting raw files.
- Synthetic amplification must never be presented as real mainnet data.
